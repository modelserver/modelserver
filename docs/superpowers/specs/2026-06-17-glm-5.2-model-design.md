# Add glm-5.2 model support

**Date:** 2026-06-17
**Status:** Approved, ready for plan

## Goal

Register `glm-5.2` (Zhipu / Z.AI) in the model catalog and across every
subscription plan and rate-limit policy, so it (a) shows up in the admin
model picker, (b) bills correctly through the extra-usage path at the
official API rate, and (c) carries a calibrated subscription discount on
every existing plan (mirroring the gpt-5.4-mini / gpt-5.4-nano setup).
All plans take effect on next deployment.

The change is rate-only and catalog-only: no router, parser, or provider
code needs to be touched. Modelserver routes by model-name string, so once
the catalog entry exists and an upstream's `supported_models` is updated
by an admin in the UI, traffic flows automatically.

## Rate calculations

Official API price (Z.AI international, per 1M tokens):

| model | input | cached_input | output |
|---|---|---|---|
| glm-5.2 | $1.40 | $0.26 | $4.40 |

GLM-5.2 has a flat 1M-context price (no long-context tier), confirmed
against the Z.AI overview docs and public reporting. `cache_creation` is
not billed as a separate event — cache misses are billed as ordinary
input.

**Catalog rate = API price ÷ 7.5** (the project-wide USD conversion,
documented in `001_init.sql:240`). This is what `computeExtraUsageCostFen`
reads, so it MUST equal the official rate:

| field | math | value |
|---|---|---|
| input_rate | 1.40 / 7.5 | 0.187 |
| output_rate | 4.40 / 7.5 | 0.587 |
| cache_creation_rate | n/a | 0 |
| cache_read_rate | 0.26 / 7.5 | 0.035 |

**Plan rate = catalog × 0.1** (chosen for glm-5.2 specifically — slightly
less aggressive than the 0.0654 multiplier used for gpt-5.5 / gpt-5.4-mini
/ gpt-5.4-nano, reflecting GLM's lower headline API price and the desire
to keep subscription burn-down comparable to other chat-tier models on a
per-prompt basis):

| field | math | value |
|---|---|---|
| input_rate | 0.187 × 0.1 | 0.0187 |
| output_rate | 0.587 × 0.1 | 0.0587 |
| cache_creation_rate | – | 0 |
| cache_read_rate | 0.035 × 0.1 | 0.0035 |

Sanity-check against neighbours: glm-5.2 plan output 0.0587 sits between
gpt-5.5 (0.261) and gpt-5.4-mini (0.0175) — the right neighbourhood for a
$4.40/M output model.

No `long_context` block — Z.AI publishes one flat rate across the full
1M context window, unlike OpenAI's 272K threshold.

## File changes

### 1. New migration `internal/store/migrations/046_add_glm_5_2.sql`

Pattern: lifted from `045_add_gpt_5_4_mini_nano.sql` (catalog INSERT +
per-plan seed + per-policy seed), with the long_context block removed
and one catalog row instead of two.

- `INSERT INTO models … ON CONFLICT (name) DO NOTHING` — single row,
  publisher `zhipu`, status `active`, empty aliases, metadata carrying
  `{"context_window": 1000000, "category": "chat"}`.
- `UPDATE plans SET model_credit_rates = jsonb_set(model_credit_rates,
  '{glm-5.2}', <plan_rate_json>, true) WHERE NOT (model_credit_rates ?
  'glm-5.2')` — idempotent, preserves any operator-set override.
- Same `UPDATE` pattern against `rate_limit_policies`.

All three statements set `updated_at = NOW()`.

### 2. `internal/admin/handle_utilization_analysis.go`

Add one entry to `utilizationAnalysisBaseRates` using the **plan rate**
(this table is consumed by the OLS regression for subscription-window
utilization, not the API-price path). No `LongContext` field. Place it
after the `gpt-5.x` block, before the codex / claude rows or wherever the
existing alphabetical-ish order lands it.

```go
"glm-5.2": {InputRate: 0.0187, OutputRate: 0.0587, CacheCreationRate: 0, CacheReadRate: 0.0035},
```

### 3. `dashboard/src/pages/admin/PlansPage.tsx`

Add one entry to `DEFAULT_MODEL_CREDIT_RATES` using the **plan rate**
(admins editing a plan get this prefilled when adding the model). No
`long_context` block.

```ts
"glm-5.2": {
  input_rate: 0.0187,
  output_rate: 0.0587,
  cache_creation_rate: 0,
  cache_read_rate: 0.0035,
},
```

## Out of scope

- **Router/parser/provider code** — model-name routing is dynamic; the
  existing OpenAITransformer / AnthropicTransformer both consume
  `upstream.BaseURL` directly with no provider-specific URL constants.
  Either Z.AI's OpenAI-compat endpoint or its Anthropic-compat endpoint
  can back the model with no transformer change (same precedent as
  deepseek-v4 in spec `2026-04-26-deepseek-v4-models-design.md`).
- **Upstream / route / upstream-group seeding** — operator creates these
  in the admin UI after deploy, since they require a real API key.
- **`035_seed_catalog_default_credit_rates.sql` comment** — its
  "intentionally left without default rate" list mentions `glm-*`;
  technically stale after this migration (glm-5.2 now has one, other
  glm-* still don't). Migration 046's header comment will mention this so
  readers don't trip over it. No edit to 035 itself.
- **GLM-5.2 variants** (`glm-5.2-air`, etc.) — not requested; add later if
  Zhipu publishes them.
- **CNY-based pricing** — we use Z.AI's international USD pricing because
  bigmodel.cn doesn't publish a stable per-token rate for glm-5.2 yet. If
  CNY billing is later required, a follow-up migration can replace these
  numbers.

## Idempotency and rollback

- Migration is gated by `schema_migrations` (runs once per DB).
- `ON CONFLICT (name) DO NOTHING` on the catalog INSERT — re-runs are
  no-ops.
- `NOT (model_credit_rates ? 'glm-5.2')` guards on UPDATEs — preserves
  any operator-set custom rate set between deploy and re-run.
- Forward-only; no down migration. To remove, write a follow-up
  migration deleting the catalog row + stripping the key from
  `plans.model_credit_rates` and `rate_limit_policies.model_credit_rates`.

## JSON payload (verbatim, for reference)

**Catalog `default_credit_rate`:**

```json
{"input_rate":0.187,"output_rate":0.587,"cache_creation_rate":0,"cache_read_rate":0.035}
```

**Plan / policy `model_credit_rates` entry:**

```json
{"input_rate":0.0187,"output_rate":0.0587,"cache_creation_rate":0,"cache_read_rate":0.0035}
```

## Risks

- **Cache field name on Anthropic-compat path.** If Z.AI's
  Anthropic-compat endpoint emits cache-read tokens under a non-standard
  field, `cache_read_tokens` will read 0 and we under-bill cache hits —
  billing-favorable but worth knowing. Same known caveat as deepseek-v4.
- **Auth header.** `AnthropicTransformer` sets `x-api-key`; if Z.AI's
  Anthropic-compat endpoint rejects that, switch the upstream to
  `provider="openai"` and route to the same model.
- **API price changes.** Z.AI has been adjusting pricing; the $1.40 /
  $4.40 / $0.26 used here is the international rate as of 2026-06-17.
  Future shifts require a follow-up migration.
- **0.1 subscription multiplier.** Chosen for glm-5.2 specifically (not
  the 0.0654 calibrated against gpt-5.5 Max-20x windows). If operations
  later wants GLM cheaper/dearer inside subscriptions, swap the
  multiplier in a follow-up — catalog and extra-usage are unaffected.

## Verification checklist

After implementation:

1. `go test ./internal/store/... -count=1` — migration tests pass
   (especially any migration-ordering test that loads the full chain).
2. `go test ./internal/admin/... -count=1` —
   `handle_utilization_analysis` tests still pass.
3. `cd dashboard && npm run typecheck && npm run lint` — TS additions
   compile.
4. `go build ./...` — whole project builds.
5. Manually apply the migration to a scratch DB, then
   `SELECT name, default_credit_rate FROM models WHERE name = 'glm-5.2';`
   to confirm the catalog row with the expected JSONB.
6. `SELECT slug, model_credit_rates->'glm-5.2' FROM plans;` — should
   match the plan-rate JSON on every row.
7. `SELECT name, model_credit_rates->'glm-5.2' FROM rate_limit_policies;`
   — same.
